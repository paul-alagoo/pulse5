# Pulse5 v0.1 Data Capture 阶段拆分

Pulse5 v0.1 的目标很明确：

**只做资料采集，不做预测、不下单、不连接钱包。**

这个版本要先把 Polymarket BTC 5 分钟 Up/Down 市场、盘口资料、BTC 价格资料稳定存下来，让之后可以 replay 和分析。

---

## Phase 1: Storage Repositories

目标：让程序可以稳定写入数据库。

要做：

- 建立 PostgreSQL client / pool
- 实作 repository 层
- 支援写入这些表：
  - `markets`
  - `raw_events`
  - `book_snapshots`
  - `btc_ticks`
- 每个表都要有 insert 或 upsert 方法
- 写测试，确认资料可以写入并读回

完成标准：

- 数据库 schema 可以从零建立
- 每个 repository 的 round-trip test 通过

---

## Phase 2: Market Discovery

目标：找到当前和接下来的 BTC 5 分钟 Up/Down 市场。

要做：

- 根据时间生成市场 slug，例如：
  - `btc-updown-5m-{timestamp}`
- 请求 Polymarket Gamma API
- 解析市场资料：
  - `event_id`
  - `market_id`
  - `slug`
  - `question`
  - `condition_id`
  - `up_token_id`
  - `down_token_id`
  - `start_time`
  - `end_time`
- 把市场写入 `markets` 表
- 如果 API 暂时没有资料，要重试，不能让 collector 崩掉

完成标准：

- discovery loop 跑 15 分钟
- 至少写入 3 个有效市场
- 每个市场都有有效的 Up / Down token IDs

---

## Phase 3: CLOB WebSocket Ingestion

目标：连接 Polymarket CLOB WebSocket，抓市场盘口资料。

要做：

- 订阅已经发现的 Up / Down token IDs
- 开启 `custom_feature_enabled: true`
- 接收这些事件：
  - `book`
  - `price_change`
  - `best_bid_ask`
  - `new_market`
  - `market_resolved`
- 所有原始事件都写入 `raw_events`
- 把 `book` 和 `best_bid_ask` 转成 `book_snapshots`
- 未知事件也要保存，不能崩溃
- WebSocket 断线后要自动重连

完成标准：

- 对一个真实市场跑 5 分钟
- 数据库中有 raw events
- 数据库中有 normalized book snapshots
- 手动断线后可以恢复

---

## Phase 4: RTDS Price Ingestion

目标：连接 Polymarket RTDS，抓 BTC 价格资料。

要做：

- 订阅 Binance BTC 价格：
  - `crypto_prices`
  - `btcusdt`
- 订阅 Chainlink BTC/USD 价格：
  - `crypto_prices_chainlink`
  - `btc/usd`
- 每 5 秒维持 ping
- WebSocket 断线后自动重连
- 原始 payload 写入 `raw_events`
- 正规化价格写入 `btc_ticks`
- 计算 `latency_ms`

完成标准：

- collector 跑 5 分钟
- Binance price 和 Chainlink price 都有资料
- 数据库中有 normalized BTC ticks
- 日志能看到 latency 情况

---

## Phase 5: Collector Loop

目标：把 discovery、CLOB ingestion、RTDS ingestion、storage 串起来。

要做：

- 启动时连接数据库
- 启动 market discovery loop
- 新市场出现时订阅 CLOB WebSocket
- 启动 RTDS price ingestion
- 把 raw events 和 normalized rows 写入数据库
- 每 30 秒输出健康状态：
  - 当前市场数量
  - WebSocket 状态
  - raw events 写入数量
  - normalized rows 写入数量
  - 每个资料源最后收到事件的时间
- 支援 graceful shutdown

完成标准：

- `pnpm dev:collector` 可以持续运行
- WebSocket 掉线后可以恢复
- 每 30 秒有健康日志
- 不交易、不下单、不需要钱包

---

## Phase 6: Replay-Ready Verification

目标：确认现在存下来的资料以后可以用来 replay。

要做：

- 新增 `research/replay/README.md`
- 说明如何从数据库重建一个市场
- 检查资料是否完整：
  - market metadata
  - raw CLOB events
  - normalized book snapshots
  - raw RTDS events
  - normalized BTC ticks
  - source timestamp
  - receive timestamp

完成标准：

- collector 跑 24 小时
- 能从数据库重建至少 50 个已结束的 BTC 5 分钟市场

---

## 建议执行顺序

1. Storage repositories
2. Market discovery
3. CLOB WebSocket ingestion
4. RTDS price ingestion
5. Collector loop
6. Replay-ready verification

---

## v0.1 安全边界

v0.1 不做以下事情：

- 不预测市场
- 不产生交易信号
- 不做 paper trading
- 不做 live trading
- 不连接钱包
- 不处理 private key
- 不下单

v0.1 的成功标准只有一个：

**稳定采集、储存、并能 replay BTC 5 分钟 Up/Down 市场资料。**
