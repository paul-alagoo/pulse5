# v0.2.2 Signal Engine Redesign：重做 5 分鐘 BTC 市場的 Estimator

## 狀態

v0.3 目前被阻擋。

這是一個 v0.2.x redesign 任務，不是 v0.3 promotion 任務。

## 背景

v0.2.1 replay 與 diagnostics 顯示，現有 signal engine 無法產生足夠的 accepted / labeled samples 來做 calibration。

7-day unchanged dry-run 結果：

- markets.replayReady: `340`
- states.built: `20,060`
- signals.total: `20,060`
- signals.accepted: `2`
- acceptedRate: 約 `0.01%`
- total labeledAccepted: `2`
- 所有 key p_win bucket 都低於每 bucket `30` labeledAccepted 的最低樣本門檻

Diagnostic sensitivity sweep 結果：

- 放寬 `minDistanceBps` 後，accepted signals 從 `2` 增加到 `17`
- 但新增樣本的勝率接近隨機：
  - overall: `8 win / 9 loss`
  - `p_win 0.50-0.60` bucket: `6 win / 9 loss`
- 這表示 current distance-based estimator 對 5-minute BTC markets 不是可靠的 win-probability proxy

目前結論：

- 這不是 calibration success
- 這不足以推進 v0.3
- 這不應該用 threshold tuning 修掉
- v0.2.2 應該 redesign estimator 與 gate structure

## 問題描述

v0.2.1 signal engine 主要依賴 BTC 現價與 `price_to_beat` 的距離，也就是 raw distance bps，作為 edge proxy。

對 5 分鐘 BTC 短窗口市場來說，raw distance alone 看起來不足夠，原因是：

1. BTC 大多時間都待在目前 `minDistanceBps` gate 內，導致幾乎沒有 accepted signals。
2. 放寬 distance gate 雖然能增加 accepted density，但沒有帶來可靠勝率。
3. estimated `p_win` 與 realized outcome 沒有明顯對齊。
4. 多個獨立 hard gates 讓 accepted signals 變得稀有，也可能掩蓋 estimator 本身的弱點。

核心問題比較像是 estimator 不適用，而不是 thresholds 太保守。

## 目標

建立 v0.2.2 signal-engine redesign，並用同一個 7-day replay window 做誠實評估。

v0.2.2 應該：

- 替換或大幅修改 raw distance-only probability estimator
- 將 distance 視為其中一個 feature，而不是整個 model
- 納入 short-window BTC regime context
- 避免讓多個 strategy hard gates 同時獨立 veto signal
- 產生足夠的 accepted / labeled samples，讓 calibration 可以被評估
- 保留 v0.2.1 artifacts 作為 baseline evidence

## 非目標

不要做以下事情：

- 不要推進 v0.3
- 不要新增 paper execution simulation
- 不要為了增加 accepted count 而直接調 production config thresholds
- 不要把 diagnostic sweep 結果宣稱為 strategy effectiveness
- 不要把 diagnostic-only sweep 結果當成 promotion evidence
- 不要刪除或重寫 v0.2.1 reports

## v0.2.2 設計方向

### 1. 新 Estimator

引入 v0.2.2 estimator，使用多個 short-window features：

- signed distance from `price_to_beat`
- absolute distance in bps
- distance normalized by recent BTC volatility
- time remaining
- recent BTC momentum / slope
- spread and entry price
- 如果現有資料已可取得，可加入 market book / liquidity features

不要再使用：

```text
p_win = f(distanceBps)
```

應改往：

```text
p_win = f(
  signedDistanceBps,
  normalizedDistance,
  timeRemainingMs,
  recentMomentumBps,
  realizedVolatilityBps,
  entryPrice,
  spread
)
```

v0.2.2 第一版仍可使用 deterministic heuristic，不一定要直接做 trained model。

但它必須被設計成可以在 replay 後檢查 estimated probability 與 realized outcome 是否對齊。

### 2. Gate Structure Redesign

將現有「所有 hard gates 都必須通過」的策略，改成更清楚的兩階段決策。

第一階段：Data-quality gates。

這些仍然應該是 hard gates：

- missing BTC tick
- stale BTC tick
- missing `price_to_beat`
- missing order book
- spread too wide
- invalid market window

第二階段：Strategy / EV decision。

這些不應該各自作為獨立 hard veto，而應該成為 continuous scoring inputs：

- distance
- estimated probability
- entry price
- expected value
- time remaining

最終 decision 應該主要基於 estimated EV：

```text
estimatedEv = estimatedPWin - entryAsk
```

signal 只有在 data quality 足夠，且 estimated EV 足夠好時才 accepted。

但 distance 不應該在 combined estimate 已經有利時，仍單獨 veto 一筆 trade。

### 3. Versioned Engine Behavior

v0.2.1 必須維持可重現。

v0.2.2 應該用 versioned behavior 加入，例如：

```text
signalEngineVersion: "v0.2.1" | "v0.2.2"
```

或新增獨立 estimator module，由 replay diagnostics / replay runner 指定使用。

replay report 必須清楚標示是哪個 engine version 產出的結果。

### 4. Replay Comparison

v0.2.2 必須使用同一個 baseline window：

```text
from=2026-04-21T10:28:36.816Z
to=2026-04-28T10:28:36.816Z
step-ms=5000
DB=pulse5tester
```

需要直接和 v0.2.1 比較：

- accepted count
- acceptedRate
- total labeledAccepted
- labeledAccepted by p_win bucket
- realized win/loss by bucket
- rejection reason distribution
- estimated vs realized win-rate

## 驗收標準

v0.2.2 不可被視為 v0.3-ready，除非全部條件都成立：

- `bucket_0_60_to_0_70` 至少有 `30` labeledAccepted
- `bucket_0_70_to_0_80` 至少有 `30` labeledAccepted
- key buckets 的 realized win-rate 大致落在 estimated probability range 的 `±5pp` 內
- accepted density 足以讓 calibration 有資訊量
- reports 必須明確說明：v0.2.2 在完成 v0.3 paper execution simulation 前，仍然不是 v0.3-ready

如果 v0.2.2 增加了 density，但 realized win-rate 仍接近 50% 或與 p_win 明顯不對齊，代表 estimator redesign 尚未完成。

## 最終決策規則

v0.2.2 replay 後：

- 如果 calibration buckets 有足夠樣本，且 realized win-rate 與 estimated p_win 對齊：可以進入正式 v0.3 paper execution simulation planning
- 如果 density 改善但 calibration 不佳：繼續 estimator redesign
- 如果 density 仍然太低：重做 feature set 或 market selection assumptions
- 不要從 v0.2.1 或 diagnostic sweep results 推進 v0.3
