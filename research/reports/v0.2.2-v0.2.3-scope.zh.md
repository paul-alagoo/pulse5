# v0.2.2 / v0.2.3 Scope 拆分

## 拆分結論

v0.2.2 和 v0.2.3 應該切在「模型設計是否已被預先註冊」這條線上。

- **v0.2.2**：只做 estimator 設計凍結、feature 定義、測試骨架、版本化接口設計。不得用 replay 結果反覆調整 estimator。
- **v0.2.3**：只實作 v0.2.2 已凍結的設計，並在 holdout window 上做一次誠實評估。失敗就報告失敗，不在同一任務內調參。

這樣拆的目的，是避免 v0.2.2 變成「把 threshold tuning 換成 estimator tuning」的 in-sample overfit。

---

## v0.2.2 Scope：Estimator Design Freeze

### 目標

建立 v0.2.2 estimator redesign 的可審查規格，先把模型形式、feature 視窗、參數來源與 evaluation protocol 釘死，再進入任何 replay 評估。

v0.2.2 的產出不是「證明策略有效」，而是「定義一個不能在 evaluation 後隨手修改的 estimator」。

### 必做項目

1. 建立 `research/reports/v0.2.2-estimator-design-note.md`
   - 明確定義 `pUp = P(BTC close > price_to_beat | features)`。
   - 明確定義 binary market 假設：`pDown = 1 - pUp`。
   - 明確定義 estimator 形式，例如固定 logistic heuristic：

     ```text
     pUp = sigmoid(
       beta0
       + beta1 * signedDistanceBps
       + beta2 * recentMomentumBps
       + beta3 * log(timeRemainingMs)
       + beta4 * realizedVolatilityBps
       + beta5 * normalizedDistance
     )
     ```

   - 明確列出每個 beta 的來源：固定 heuristic、fit window 估計、或人工預註冊常數。
   - 明確規定：beta 不得根據 holdout evaluation report 反覆修改。

2. 定義所有 v0.2.2 features
   - `signedDistanceBps`
   - `absoluteDistanceBps`
   - `recentMomentumBps`
   - `realizedVolatilityBps`
   - `normalizedDistance`
   - `timeRemainingMs`
   - `entryAsk`
   - `spread`
   - `estimatedEv`

3. 釘死 feature 視窗與單位
   - `recentMomentumBps`：以 target timestamp `t` 往回看固定視窗，例如 60 秒；使用 visible ticks only。
   - `realizedVolatilityBps`：以 target timestamp `t` 往回看固定視窗，例如 180 秒；用 log returns 的 standard deviation 轉 bps。
   - `normalizedDistance`：明確規定 normalization 公式，例如：

     ```text
     normalizedDistance = signedDistanceBps / max(realizedVolatilityBps, volatilityFloorBps)
     ```

   - 明確規定最少 tick 數；不足時 estimator 應回傳不可估計或使用預先定義 fallback，不可臨場猜測。

4. 釘死 deterministic BTC source fallback
   - 優先使用 Chainlink visible ticks。
   - 只有當 Chainlink tick 缺失或超過 staleness threshold 時，才使用 Binance visible ticks。
   - fallback rule 必須 deterministic，且不得讀取 `receive_ts > t` 的資料。

5. 定義 pUp 邊界
   - `pUp` 必須 clamp，例如：

     ```text
     pUp = clamp(rawPUp, 0.02, 0.98)
     ```

   - clamp 值必須在 v0.2.2 設計文件中固定。

6. 定義 v0.2.3 evaluation protocol
   - Baseline full window 仍保留作為對照：

     ```text
     2026-04-21T10:28:36.816Z 到 2026-04-28T10:28:36.816Z
     ```

   - 若 estimator 有任何 fitted parameter，必須切分：

     ```text
     fit window:      2026-04-21T10:28:36.816Z 到 2026-04-25T10:28:36.816Z
     holdout window:  2026-04-25T10:28:36.816Z 到 2026-04-28T10:28:36.816Z
     ```

   - 若 estimator 完全使用預註冊 heuristic constants，仍必須把「沒有 fit」寫清楚，並禁止根據 holdout 結果改常數。

7. 建立測試骨架
   - v0.2.1 regression tests：確保舊 engine output 不變。
   - v0.2.2 estimator tests：先寫預期行為，不需要完整 implementation。
   - no-lookahead tests：確保 feature extraction 不會使用 future ticks。

### 明確不做

- 不跑 v0.2.2 density replay 作為 calibration 證據。
- 不產出 `research/reports/v0.2.2-density-7d.json` 作為策略評估。
- 不修改 production thresholds 來增加 accepted count。
- 不新增 paper execution simulation。
- 不新增 orders、simulated_orders、fills、positions、wallet、signer。
- 不改寫或刪除 v0.2.1 reports。
- 不在看到 replay 結果後修改 beta、feature window、EV threshold。

### v0.2.2 交付物

- `research/reports/v0.2.2-estimator-design-note.md`
- `research/reports/v0.2.2-v0.2.3-scope.zh.md`
- v0.2.2 estimator/type/interface skeleton
- focused failing tests 或 test skeleton
- 一段明確 handoff：哪些設計已 freeze，哪些問題需要人工 review

### v0.2.2 驗收標準

- estimator 形式已明確。
- 每個 feature 的視窗、單位、fallback、invalid rule 已明確。
- fit window / holdout window 已明確。
- `pUp` 語意、clamp、`pDown = 1 - pUp` 假設已明確。
- v0.2.1 reproducibility 未被破壞。
- 文件明確禁止在 v0.2.3 holdout evaluation 後同任務內調參。

## v0.2.3 Scope：Frozen Estimator Implementation + Holdout Evaluation

### 目標

實作 v0.2.2 已凍結的 estimator 與 versioned engine，然後在 holdout window 上做一次 replay 評估。

v0.2.3 的任務不是「把報告調到通過」，而是「測量凍結設計是否真的比 v0.2.1 更可評估、更接近 calibration」。

### 必做項目

1. 實作 versioned engine
   - 新增：

     ```text
     SignalEngineVersion = "v0.2.1" | "v0.2.2"
     ```

   - v0.2.1 行為必須保持可重現。
   - v0.2.2 透過明確參數指定，不取代 default engine。
   - replay report 必須 carry：

     ```text
     config.signalEngineVersion
     ```

2. 實作 frozen v0.2.2 estimator
   - 僅依照 v0.2.2 design note 實作。
   - 不得因第一次 replay 結果不好而改 beta、feature 視窗、fallback rule、clamp、EV threshold。
   - `estimatedEv` 使用：

     ```text
     upEv = pUp - upBestAsk
     downEv = (1 - pUp) - downBestAsk
     ```

   - 選擇 EV 較高的一邊作為 candidate side。

3. 實作兩階段 gate structure
   - Stage 1：data-quality hard gates
     - missing BTC tick
     - stale BTC tick
     - missing `price_to_beat`
     - missing order book / ask
     - spread too wide
     - invalid market window
     - feed gap too large
   - Stage 2：strategy / EV decision
     - distance、entry price、time remaining、estimated probability 不再各自 hard veto。
     - 最終只以 selected side 的 `estimatedEv >= v0.2.2MinEstimatedEv` 決定是否接受。
   - v0.2.2 的 `minEstimatedEv` 必須是版本專屬設定，不可默默沿用 v0.2.1 config。

4. 實作 replay CLI 支援
   - 加入：

     ```text
     --engine-version=v0.2.1 | v0.2.2
     ```

   - default 可維持 v0.2.1，以保護既有 replay。
   - `--persist` 若未指定 `--engine-version`，必須 fail fast。
   - v0.2.3 初期以 dry-run 為主；不要把多版本 signals 寫入 DB，除非另開 schema task。

5. 產出 holdout report
   - 必須使用 `pulse5tester`。
   - fit window 和 holdout window 不得重疊。
   - 主要 calibration 判斷只看 holdout window。
   - full 7-day window 可以產出作為 descriptive comparison，但不得當成 calibration 通過證據。

6. 報告必須包含
   - v0.2.1 vs v0.2.2 accepted count
   - acceptedRate
   - total labeledAccepted
   - labeledAccepted by p_win bucket
   - realized win/loss by bucket
   - estimated vs realized win-rate
   - rejection reason distribution
   - v0.2.1 vs v0.2.2 rejection reason distribution shift
   - 是否符合 key buckets 最低樣本門檻
   - 是否符合 realized win-rate 在 estimated probability range 的 +/-5pp 內

### Fail-fast rule

v0.2.3 必須遵守：

- 如果第一次 frozen estimator replay 後 `accepted = 0`，立即停止。
- 如果 accepted density 有改善，但 holdout calibration 明顯不對齊，立即停止。
- 如果 key buckets 樣本不足，立即停止。
- 停止後只寫報告與原因，不在同一 task 內修改 estimator。

### 明確不做

- 不根據 holdout report 調 beta。
- 不根據 holdout report 改 feature window。
- 不根據 holdout report 改 `v0.2.2MinEstimatedEv`。
- 不把 full 7-day in-sample result 當成 calibration 證據。
- 不把 diagnostic sweep 當成 promotion evidence。
- 不推進 v0.3。
- 不新增 paper execution simulation。
- 不新增 persistence schema，除非另開明確 v0.2.4 或 schema-only task。

### v0.2.3 交付物

- versioned signal-engine implementation
- frozen estimator implementation
- feature extraction implementation
- focused unit tests
- v0.2.1 regression tests
- replay CLI engine-version tests
- holdout replay JSON report，例如：

  ```text
  research/reports/v0.2.3-holdout-density.json
  ```

- full comparison summary，例如：

  ```text
  research/reports/v0.2.3-holdout-summary.md
  ```

### v0.2.3 驗收標準

v0.2.3 只有在以下條件全部成立時，才可以說「可以進入 v0.3 paper execution simulation planning」：

- holdout `bucket_0_60_to_0_70` 至少有 `30` labeledAccepted。
- holdout `bucket_0_70_to_0_80` 至少有 `30` labeledAccepted。
- key buckets realized win-rate 大致落在 estimated probability range 的 `+/-5pp` 內。
- accepted density 足以讓 calibration 有資訊量。
- report 明確說明：即使 v0.2.3 通過，也仍然不是 live-ready，只能進入 v0.3 paper execution simulation planning。

若任何一項失敗，結論必須是：

```text
v0.2.3 frozen estimator evaluation failed; continue estimator redesign in a new scoped task.
```

## 建議執行順序

1. v0.2.2：只寫 design note、freeze estimator、寫測試骨架，停止等待 review。
2. 人工 review v0.2.2 design note。
3. v0.2.3：依照 frozen design 實作，不調參。
4. v0.2.3：跑 holdout replay，一次性產出 report。
5. 根據 report 做決策：
   - calibration 有樣本且對齊：進入 v0.3 paper simulation planning。
   - density 有改善但 calibration 不佳：另開下一個 estimator redesign task。
   - density 仍太低：另開 feature set / market selection 假設重審 task。
