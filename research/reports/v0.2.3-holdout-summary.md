Shape change is consistent with the v0.2.2 design:

- `BTC_TOO_CLOSE_TO_PRICE_TO_BEAT` is dropped (distance becomes a
  feature in the estimator, not a hard veto) — the histogram entry
  reaches zero, as required.
- `ENTRY_PRICE_TOO_EXPENSIVE` is no longer raised — final acceptance
  is the EV gate alone (this PR's P1 fix). Confirms the code change
  reaches the runtime path.
- `NO_EDGE` shrinks from "almost everything" to ~14 % of states. The
  data-quality gates (`STALE_*`, `*_AGE`, `*_GAP`) carry the same
  v0.2.1 numbers — the design preserves them, and `pulse5tester`
  evidently has no stale-feed instances in this window.
- `SPREAD_TOO_WIDE` rises slightly from 35 → 38: v0.2.2 evaluates spread
  on the *selected* side, which under EV-side selection differs from
  v0.2.1's sign-of-distance side selection on a handful of states.

## Calibration verdict

The scope acceptance check is whether each key bucket's realized win-rate
falls inside that bucket's estimated probability range, widened by ±5 pp.
The first key bucket fails that check:

- `0.50 ≤ p < 0.60`: realized 0.4245, below the acceptable 0.45 to 0.65 range by **2.6 pp**.
- `0.60 ≤ p < 0.70`: realized 0.5638, inside the acceptable 0.55 to 0.75 range.
- `0.70 ≤ p < 0.80`: realized 0.6549, inside the acceptable 0.65 to 0.85 range.
- `p ≥ 0.80`: realized 0.7869, inside the acceptable ≥0.75 range.

Because every required acceptance criterion must pass, the failed
`0.50 ≤ p < 0.60` calibration bucket is sufficient to reject v0.2.3.
Per design-note §9 (no-post-evaluation-tuning rule) and §"Fail-fast
rule", the only acceptable response is to stop, document, and open a
fresh estimator-redesign task.

## Conclusion

v0.2.3 frozen estimator evaluation failed; continue estimator redesign in a new scoped task.
