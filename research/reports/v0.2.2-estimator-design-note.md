# v0.2.2 Estimator Design Note (Frozen)

## Status

**Design freeze.** This document defines the v0.2.2 signal-engine estimator
*before* any replay or holdout evaluation has been run.

v0.2.2 itself ships only the design freeze and limited type / interface
scaffolding. The numerical body of the estimator, replay calibration, and
holdout evaluation are explicitly **out of scope** for v0.2.2 and are owned
by **v0.2.3** (see `research/reports/v0.2.2-v0.2.3-scope.zh.md`).

The point of freezing the design here is to prevent v0.2.2 from quietly
turning into "threshold tuning replaced by estimator tuning". Once v0.2.3
runs the holdout replay, no parameter in this document may be changed in
the same task without a fresh, separately-scoped redesign.

## Source of Truth

This note is the authoritative spec the v0.2.3 implementation must match.
It supersedes implementation hints in
`research/reports/v0.2.2-redesign-issue-spec.zh.md`. If the two disagree,
this note wins; the redesign issue spec is the *motivation* document, not
the implementation contract.

## 1. Probability Semantics

The v0.2.2 estimator computes:

```text
pUp   = P(BTC close > price_to_beat | features at engine ts)
pDown = 1 - pUp
```

Notes:

- `pUp` is conditional on the **observed** features at engine timestamp `t`
  only. No data with `receive_ts > t` may be consulted.
- The market is treated as binary. There is no third "tie / push" outcome
  in the estimator; tie handling is a market-resolution concern for the
  outcome labeler, not the estimator.
- `pUp` and `pDown` always sum to 1 by construction.

## 2. Frozen Estimator Form

v0.2.2 freezes the estimator as a deterministic logistic heuristic over a
fixed feature vector:

```text
rawPUp = sigmoid(
    beta0
  + beta1 * signedDistanceBps
  + beta2 * recentMomentumBps
  + beta3 * log(timeRemainingMs)
  + beta4 * realizedVolatilityBps
  + beta5 * normalizedDistance
)
pUp = clamp(rawPUp, 0.02, 0.98)
```

where `sigmoid(x) = 1 / (1 + exp(-x))`.

**Why this form is frozen now:**

- Deterministic. Live and replay must agree bit-for-bit on every decision,
  exactly as v0.2.1 does.
- Pure. No I/O, no clock, no DB; same purity contract as v0.2.1's
  `generateSignal`.
- Linear-in-features. Easy to inspect by hand; easy to reproduce; cheap
  enough for tick-rate scoring.
- The form is **non-negotiable** in v0.2.3. Adding or removing a feature
  is a redesign, not an implementation detail.

## 3. Beta / Weight Source

All `beta*` values are **pre-registered constants**. v0.2.2 has *no fitted
parameters*.

- `beta0..beta5` are hand-picked heuristic constants chosen so the
  estimator (a) reproduces v0.2.1-style "BTC clearly above price_to_beat
  → pUp > 0.5" behavior and (b) gives non-trivial sensitivity to
  momentum and volatility.
- The exact numeric values of `beta0..beta5` are reserved for the v0.2.3
  implementation PR. They are committed *before* the holdout replay is
  run, and may not be changed in response to the holdout report.
- Because no parameter is fit on data, v0.2.2 has **no in-sample fit
  step**. The fit/holdout split below applies *only* if a future
  redesign introduces fitted parameters.

**Rule:** If at v0.2.3 implementation time someone wants to read replay
output and adjust `beta*`, that is a *new redesign task*, not part of
v0.2.3.

## 4. Features, Windows, Units

All features are extracted from data with `receive_ts ≤ engine ts t`. No
look-ahead.

| Feature | Definition | Window | Units |
|---|---|---|---|
| `signedDistanceBps` | `(btcPrice - priceToBeat) / priceToBeat * 10_000` | instantaneous at `t` | bps, signed |
| `absoluteDistanceBps` | `abs(signedDistanceBps)` | instantaneous at `t` | bps, ≥ 0 |
| `recentMomentumBps` | `(btcPrice(t) - btcPrice(t - 60_000)) / btcPrice(t - 60_000) * 10_000` | last 60 s of visible BTC ticks | bps, signed |
| `realizedVolatilityBps` | `stdev(log returns of BTC ticks in last 180 s) * 10_000` | last 180 s of visible BTC ticks | bps, ≥ 0 |
| `normalizedDistance` | `signedDistanceBps / max(realizedVolatilityBps, volatilityFloorBps)` | derived | dimensionless, signed |
| `timeRemainingMs` | `market.endTime - t` | instantaneous at `t` | ms |
| `entryAsk` | selected side's `bestAsk` (UP if `signedDistanceBps > 0`, else DOWN) | instantaneous at `t` | Yes/No price (0..1) |
| `spread` | selected side's `bestAsk - bestBid` | instantaneous at `t` | Yes/No price (0..1) |
| `estimatedEv` | `selectedP - entryAsk`, where `selectedP = pUp` for UP and `1 - pUp` for DOWN | derived | Yes/No price (0..1), signed |

Constants frozen here:

- **Momentum window:** 60_000 ms.
- **Volatility window:** 180_000 ms.
- **Volatility floor (`volatilityFloorBps`):** 1.0 bps. Prevents division
  by zero / overflow when BTC has been flat across the window.
- **Minimum tick count for momentum:** 2 (need at least one prior tick
  inside the 60 s window).
- **Minimum tick count for volatility:** 3 (need ≥ 2 log returns).
- **Minimum tick count for `pUp`:** the union of the above; if either
  feature is unavailable, the estimator must report invalid (see §6).
- **`btcPrice(t - 60_000)` definition:** the BTC tick with the largest
  `receive_ts` satisfying `receive_ts ≤ t - 60_000`. If none exists in
  the visible window, momentum is invalid.

Side selection is determined by the sign of `signedDistanceBps` so that
`pUp` and `entryAsk` are evaluated consistently.

## 5. Deterministic BTC Source / Fallback

Identical contract to v0.2.1, repeated here so v0.2.3 cannot drift:

1. **Primary source:** `rtds.chainlink` ticks visible at `t`.
2. **Fallback source:** `rtds.binance` ticks visible at `t` — used **only
   when** the most recent Chainlink tick is missing or has
   `t - receive_ts > maxBtcTickAgeMs`. Threshold reuses
   `StrategyConfig.maxBtcTickAgeMs` (currently 10_000 ms).
3. The selected source must be the same one `buildMarketState` chose for
   `MarketState.btcSource`. The estimator does not re-pick.
4. Once a source is chosen for `t`, *all* features at `t` (price, momentum,
   volatility) are computed from that source's tick stream only. Mixing
   feeds inside one decision is forbidden.
5. The fallback rule is deterministic: replaying the same data twice picks
   the same source, the same ticks, and produces the same `pUp`.

## 6. Invalid-Data Behavior

The estimator must **fail closed**, not guess, when any of the following
hold at `t`:

- `priceToBeat` is null.
- BTC tick is missing or stale (per `maxBtcTickAgeMs`).
- Fewer than the minimum tick counts in §4 are available in the relevant
  window.
- `realizedVolatilityBps` resolves to a non-finite value.

Failure-closed semantics:

- The estimator returns "no estimate" — represented as `pUp = null` (or an
  equivalent "invalid" sentinel chosen by v0.2.3, but **never** silently
  defaulting to 0.5).
- The downstream signal must REJECT with a single explicit reason. v0.2.3
  selects the exact `SignalRejectionReason` value; v0.2.2 only freezes
  *that there must be one*, not invent it here. Reusing existing reasons
  (`DATA_INCOMPLETE`, `STALE_BTC_TICK`, `PRICE_TO_BEAT_MISSING`) is the
  preferred path; adding a new reason is permissible only if no existing
  one fits.
- Whatever reason v0.2.3 picks must be deterministic given the same input.

## 7. pUp Clamp Range

```text
pUp = clamp(rawPUp, 0.02, 0.98)
```

- Lower bound `0.02`, upper bound `0.98`. **Frozen.**
- Rationale: pure logistic outputs can saturate, which produces
  `estimatedEv` values that look attractive but are an artifact of
  arithmetic, not edge. The clamp is a deliberate cap on how much
  confidence the heuristic is allowed to express.
- The clamp is applied **after** the sigmoid and **before** any EV
  arithmetic.

## 8. Evaluation Protocol (owned by v0.2.3)

The 7-day baseline window remains the comparison frame:

```text
2026-04-21T10:28:36.816Z … 2026-04-28T10:28:36.816Z
```

Even though v0.2.2 has no fitted parameters, a fit/holdout split is
declared **prospectively** so any future redesign that does fit
parameters cannot quietly dodge the discipline:

- **Fit window:** `2026-04-21T10:28:36.816Z … 2026-04-25T10:28:36.816Z`
  (4 days). Reserved for future fitting; v0.2.3 does **not** fit on it.
- **Holdout window:** `2026-04-25T10:28:36.816Z …
  2026-04-28T10:28:36.816Z` (3 days). v0.2.3's calibration verdict is
  judged on this window only.

The full 7-day window may be replayed for descriptive comparison against
v0.2.1, but **must not** be used as the calibration verdict.

## 9. No-Post-Evaluation-Tuning Rule

After v0.2.3 runs the holdout replay, the following are **forbidden**
inside the same task:

- Editing `beta0..beta5`.
- Editing any feature window (`60_000`, `180_000`).
- Editing `volatilityFloorBps`.
- Editing the `pUp` clamp bounds.
- Editing the BTC source / fallback rule.
- Editing the v0.2.2-specific `minEstimatedEv` (whatever value v0.2.3
  ships as the version-specific threshold).
- Re-defining "selected side".

If holdout calibration fails, the only acceptable response is the
fail-fast rule from
`research/reports/v0.2.2-v0.2.3-scope.zh.md` §"Fail-fast rule": stop,
write a report explaining the failure, and open a *new* redesign task.

## 10. Expected Failure Modes

These are the failure modes most likely to surface in the v0.2.3
holdout. They are documented here so seeing them does **not** count as
an excuse to retune the v0.2.2 design:

1. **Density still too low.** Even with momentum / volatility features,
   5-minute BTC windows may not produce enough accepted signals to fill
   the calibration buckets. Outcome: v0.2.3 reports density failure,
   work continues in a new feature-redesign task.
2. **Density rises but calibration drifts.** Accepted-signal win rate
   sits near 50% even when `pUp` is in the 0.6–0.8 buckets. Outcome:
   v0.2.3 reports calibration failure, work continues in a new
   estimator-redesign task.
3. **Volatility floor saturation.** A long flat BTC stretch makes
   `realizedVolatilityBps` collapse to `volatilityFloorBps`, which
   inflates `normalizedDistance`. v0.2.3 must surface this in its
   report; if it dominates accepted samples, that itself is a calibration
   failure.
4. **Source flip churn.** Repeated alternation between Chainlink and
   Binance fallback in short windows can produce visible discontinuities
   in momentum/volatility. v0.2.3 must report any market where source
   flipped inside the 180 s volatility window.

## 11. Scope Confirmation

v0.2.2 (this PR / branch):

- Adds this design note.
- Adds versioned-engine *type* and *interface* scaffolding only.
- Adds a `v0.2.2` estimator entrypoint that **fails closed** with a
  `"v0.2.3 implementation pending"` error if accidentally called.
- Keeps `v0.2.1` as the default engine and preserves bit-for-bit
  reproducibility of v0.2.1 signals.
- Adds tests for: v0.2.1 regression, version/type/interface contracts,
  fail-closed stub behavior, and "default did not switch to v0.2.2".
- Marks v0.2.3 behavior tests (pUp clamp, `pDown = 1 - pUp`, feature
  extraction from recent visible BTC ticks, no-lookahead, EV side
  selection) as `it.todo` / `it.skip` with the explicit reason
  `"v0.2.3 implementation pending"`.

v0.2.2 explicitly does **not**:

- Implement the numerical body of the estimator.
- Run replay calibration or holdout evaluation.
- Tune any parameter.
- Add orders, simulated_orders, fills, positions, wallets, signers,
  executions, or any private-key surface.
- Add or modify any persistence schema or migration.
- Promote anything to v0.3.
- Add paper execution simulation.

v0.2.3 owns the frozen implementation, the holdout replay run, and the
holdout report.
